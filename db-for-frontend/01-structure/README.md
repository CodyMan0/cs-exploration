# 1단계. 구조 — 데이터의 뼈대 세우기

## 큰 그림 (비유)

데이터베이스를 **잘 정리된 회사 서류함**이라고 생각해봐요.

- 아무 종이에나 다 적으면(엑셀 한 장) → 같은 내용이 여기저기 중복되고, 하나 고치면 나머지를 다 찾아 고쳐야 함
- 잘 나눠진 서류함 → "병원 정보"는 병원 칸에, "예약 정보"는 예약 칸에, 칸끼리는 **번호표(키)**로 연결

이 "잘 나누기"가 **정규화**, 칸끼리 잇는 번호표가 **PK / FK**예요.

## 핵심 개념 미리보기

### PK (Primary Key, 기본키)
- 한 행을 **유일하게** 가리키는 값. 중복 불가, NULL 불가.
- 예: `patients.id = 42` 면 세상에 그 환자 단 한 명.
- 프론트 비유: React 리스트의 `key` prop. 그게 겹치면 화면이 꼬이듯, PK가 겹치면 데이터가 꼬여요.

### FK (Foreign Key, 외래키)
- 다른 테이블의 PK를 가리키는 값. "이 예약은 42번 환자 거"를 `patient_id = 42`로 표현.
- DB가 **무결성**을 강제: 존재하지 않는 99번 환자로 예약을 만들려 하면 **거부**해요.

### 정규화 (Normalization)
- 중복을 없애 **한 사실은 한 곳에만** 저장하는 작업.
- 1NF: 한 칸에 값 하나 (전화번호 여러 개를 콤마로 욱여넣지 않기)
- 2NF/3NF: "병원 이름" 같은 건 예약마다 반복하지 말고 병원 테이블에 한 번만

### 역정규화 & JSONB
- 정규화가 항상 정답은 아님. 자주 같이 읽는데 굳이 매번 JOIN하기 아까울 때, 혹은 **구조가 유연해야 할 때**(병원마다 다른 진료 옵션) → JSONB로 한 칸에 문서처럼 저장.
